# API-to-MCP Gateway 项目说明文档

> 写给第一次接触这个项目的人。先讲「是什么、为什么、怎么用」，再讲「怎么实现」。

---

## 第一部分：场景 —— 这个项目到底在解决什么

### 1.1 一个日常需求

你每天可能会做这些事：

- 在 GitHub 上查 Issue、创建 Pull Request
- 在 Notion 里建数据库、写文档
- 在飞书里发消息、查日历
- 在 Jira 里创建任务、查进度

这些操作本质上都是 **调 REST API**：往某个 URL 发一个 HTTP 请求，带上参数和认证信息，拿到返回结果。

### 1.2 但你不想自己去点网页

你想说一句话就搞定：

> "帮我把 GitHub 上 label 为 bug 的 Issue 全部列出来，然后每个 Issue 在 Notion 建一条记录。"

如果 AI（Claude、GPT）能直接调这些 API，这件事就能自动完成。

### 1.3 AI 怎么调 API —— MCP 协议

AI 本身不能直接发 HTTP 请求。它需要一个「中间人」—— MCP Server。

```
你说一句话  →  AI 理解意图  →  AI 调 MCP Tool  →  MCP Server 发 HTTP  →  目标 API
```

**MCP（Model Context Protocol）** 是 Anthropic 发布的一套标准协议，定义了两件事：
1. AI 怎么发现有哪些工具可用（`tools/list`）
2. AI 怎么调用这些工具（`tools/call`）

所以你想让 Claude 调 GitHub API，你需要一个 MCP Server，里面定义了一堆 tool：
- `github_list_issues` — 列出 Issue
- `github_create_issue` — 创建 Issue
- `github_list_repos` — 列出仓库
- ……

### 1.4 问题来了

GitHub 的 REST API 有 **几百个接口**。如果每个都要你手工写成 MCP tool：

```typescript
// 接口 1：列出 Issues —— 手工写
server.tool("github_list_issues", "列出仓库 Issues", {
  owner: z.string(),
  repo: z.string(),
  state: z.enum(["open", "closed"]).optional()
}, async (args) => {
  const res = await fetch(`https://api.github.com/repos/${args.owner}/${args.repo}/issues?...`);
  return res.json();
});

// 接口 2：创建 Issue —— 又手工写一遍
// 接口 3：更新 Issue —— 又一遍
// ... 还有 200 多个
```

**这个项目做的事：你不用手写。把 GitHub 的 OpenAPI 文档丢进去，自动生成全部 MCP tool。**

---

### 1.5 一句话总结

| 输入 | 输出 | 效果 |
|------|------|------|
| 一个 OpenAPI 文档（JSON/YAML） | 一个跑起来的 MCP Server | AI 可以直接调这个 API 的所有接口 |

## 第二部分：术语解释

这些词在读代码时会反复出现，这里统一解释。

### OpenAPI / Swagger

**是什么**：一种描述 REST API 的标准格式。一个 JSON 或 YAML 文件，里面写清楚了：
- 这个 API 有哪些接口（`/users`、`/issues` 等）
- 每个接口用什么 HTTP 方法（GET、POST……）
- 需要什么参数、参数是什么类型
- 请求体长什么样
- 返回什么

**长什么样**（来自项目中 `specs/petstore.yaml` 的真实例子）：

```yaml
# 这是一个宠物店 API 的描述文件
openapi: "3.0.3"                          # OpenAPI 版本号
info:
  title: "Swagger Petstore"               # API 名称
  version: "1.0.0"
servers:
  - url: "https://petstore.swagger.io/api/v3"  # API 的根地址
paths:
  /pet:                                    # 路径：操作宠物
    post:                                  # POST 方法：新增宠物
      operationId: addPet                  # 这个操作的唯一 ID
      summary: "Add a new pet to the store"
      parameters:                          # 参数列表
        - name: storeId
          in: header                       # 参数在 HTTP Header 里
          required: false
          schema:
            type: string
      requestBody:                         # 请求体结构
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [name]
              properties:
                name:
                  type: string
                status:
                  type: string
                  enum: [available, pending, sold]  # 只能填这三个值
      responses:                           # 可能的返回
        "200":
          description: "Successful operation"
        "405":
          description: "Invalid input"

  /pet/{petId}:                            # 带路径参数的接口
    get:
      operationId: getPetById
      summary: "Find pet by ID"
      parameters:
        - name: petId
          in: path                         # 参数在 URL 路径里
          required: true
          schema:
            type: integer
            format: int64

  /pet/findByStatus:                       # 按状态搜索
    get:
      operationId: findPetsByStatus
      summary: "Finds pets by status"
      parameters:
        - name: status
          in: query                        # 参数在 URL 查询字符串里
          required: true
          schema:
            type: string
```

**解读这张表**：整个文件描述了 3 个接口（`addPet`、`getPetById`、`findPetsByStatus`），每个接口的请求方式、参数位置、参数类型、返回格式都写得清清楚楚。人看得懂，程序也读得懂。

### MCP（Model Context Protocol，模型上下文协议）

**是什么**：AI 和外部工具之间的通信协议。

**为什么需要它**：AI 模型只能处理文本，不能自己发 HTTP 请求、读文件、查数据库。MCP 就是让 AI 能「用工具」的标准。

**怎么工作**：

```
AI 客户端（Claude Desktop/Code）
    │
    │  JSON-RPC over HTTP
    │
    ▼
MCP Server（本项目生成的就是这个）
    │
    │  HTTP 请求
    │
    ▼
目标 API（GitHub/Notion/飞书）
```

MCP 协议规定了几个核心方法：

| JSON-RPC 方法 | 作用 | 类比 |
|---|---|---|
| `initialize` | AI 和 Server 握手，交换能力信息 | 拨通电话 |
| `tools/list` | AI 问「你能干什么」 | 看说明书 |
| `tools/call` | AI 说「帮我干这个」 | 下指令 |
| `resources/list` | AI 问「你有什么数据」 | 看目录 |
| `resources/read` | AI 说「给我这份数据」 | 翻到那一页 |

### MCP Tool

**是什么**：MCP Server 暴露的一个一个「功能单元」。每个 Tool 有三要素：

1. **name**（名称）：`github_list_issues`
2. **description**（描述）：一段说明文字，AI 靠它理解这个工具是干嘛的
3. **inputSchema**（输入格式）：用 JSON Schema 描述这个工具需要什么参数

**实际长什么样**（项目代码转换后的结果）：

```json
{
  "name": "petstore_add_pet",
  "description": "Add a new pet to the store. Tags: pets.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "storeId": {
        "type": "string",
        "description": "storeId (header)"
      },
      "name": {
        "type": "string"
      },
      "status": {
        "type": "string",
        "enum": ["available", "pending", "sold"]
      }
    },
    "required": ["name"]
  }
}
```

**解读**：AI 读到这个 Tool 后就知道——
- 有个工具叫 `petstore_add_pet`，用来添加宠物
- 需要填 `name`（必填）和 `status`（可选，只能是 available/pending/sold 之一）
- 还有一个 header 参数 `storeId`

### Streamable HTTP Transport

**是什么**：MCP 协议的一种传输方式。AI 客户端通过 HTTP POST 请求跟 MCP Server 通信。

**为什么不用 stdio**：stdio 是进程间通信（标准输入输出），只能一个 AI 连一个 Server。HTTP 可以让多个 AI 同时连接同一个 Server。

**长什么样**：

```bash
# AI 客户端往 http://127.0.0.1:3000/mcp 发一个 POST 请求：
curl -X POST http://127.0.0.1:3000/mcp \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/list",
    "params": {}
  }'

# Server 返回：
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      { "name": "petstore_add_pet", "description": "Add a new pet...", "inputSchema": {...} },
      { "name": "petstore_get_pet_by_id", "description": "Find pet by ID", "inputSchema": {...} },
      { "name": "petstore_find_pets_by_status", "description": "Finds pets by status", "inputSchema": {...} }
    ]
  }
}
```

### RAG（Retrieval-Augmented Generation，检索增强生成）

**是什么**：让 AI 在回答问题之前，先去「查资料」再回答。

**在这个项目里的作用**：一个 API 可能有几百个接口（比如 GitHub）。AI 不可能一次性记住所有 Tool 的描述。RAG 让 AI：

1. 拿到用户问题：「我想创建 Issue」
2. 先去向量数据库搜索：「哪些接口跟创建 Issue 相关？」
3. 找到最相关的几个：`github_create_issue`、`github_update_issue`
4. 然后调用正确的 Tool

**不用 RAG 的问题**：AI 可能猜错，调用 `github_create_repo` 而不是 `github_create_issue`。

### 向量存储/Embedding

**是什么**：把一段文字转换成一串数字（向量），语义相似的文字，数字也相似。

```
"创建 Issue"    → [0.02, 0.87, -0.13, 0.45, ...]
"新增 Issue"    → [0.03, 0.85, -0.11, 0.43, ...]  ← 相似
"删除仓库"      → [-0.76, 0.12, 0.68, -0.33, ...]  ← 不相似
```

本项目用了简化版的哈希向量（`packages/rag/src/indexer.ts` 的 `hashEmbedding` 函数），做余弦相似度比对。Demo 够用。

## 第三部分：实际看一遍 —— 从输入到输出，每一步长什么样

这里用项目里真实的 `specs/petstore.yaml` 举例。

### Step 1：输入 —— OpenAPI 文档

前面已经贴过，就是 `specs/petstore.yaml`，描述了宠物店 API 的 3 个接口。

### Step 2：解析 —— `packages/core/src/parser.ts`

程序读入这个 YAML 文件，提取出 3 个 Operation 对象：

```typescript
// parser.ts 的输出 —— 3 个结构化的操作对象
[
  {
    operationId: "addPet",
    method: "POST",
    path: "/pet",
    summary: "Add a new pet to the store",
    parameters: [
      { name: "storeId", in: "header", required: false, schema: { type: "string" } }
    ],
    requestBody: {
      required: true,
      content: {
        "application/json": {
          schema: {
            type: "object",
            required: ["name"],
            properties: {
              name: { type: "string" },
              status: { type: "string", enum: ["available", "pending", "sold"] }
            }
          }
        }
      }
    }
  },
  {
    operationId: "getPetById",
    method: "GET",
    path: "/pet/{petId}",
    summary: "Find pet by ID",
    parameters: [
      { name: "petId", in: "path", required: true, schema: { type: "integer", format: "int64" } }
    ]
  },
  {
    operationId: "findPetsByStatus",
    method: "GET",
    path: "/pet/findByStatus",
    summary: "Finds pets by status",
    parameters: [
      { name: "status", in: "query", required: true, schema: { type: "string" } }
    ]
  }
]
```

### Step 3：转换 —— `packages/core/src/converter.ts`

每个 Operation 被转成一个 MCP Tool：

```typescript
// 输入：Operation 对象
// 输出：MCP Tool 定义

// addPet 变成：
{
  name: "petstore_add_pet",          // operationId 驼峰转蛇形 + 前缀
  description: "Add a new pet to the store.",
  inputSchema: {
    type: "object",
    properties: {
      "storeId": { type: "string", description: "storeId (header)" },
      "name":    { type: "string" },
      "status":  { type: "string", enum: ["available", "pending", "sold"] }
    },
    required: ["name"]
  }
}

// getPetById 变成：
{
  name: "petstore_get_pet_by_id",
  description: "Find pet by ID.",
  inputSchema: {
    type: "object",
    properties: {
      "petId": { type: "integer", format: "int64", description: "petId (path)" }
    },
    required: ["petId"]
  }
}

// findPetsByStatus 变成：
{
  name: "petstore_find_pets_by_status",
  description: "Finds pets by status.",
  inputSchema: {
    type: "object",
    properties: {
      "status": { type: "string", description: "status (query)" }
    },
    required: ["status"]
  }
}
```

**命名规则**：`{API名称}_{驼峰转蛇形}`
- `addPet` → `petstore_add_pet`
- `getPetById` → `petstore_get_pet_by_id`
- `findPetsByStatus` → `petstore_find_pets_by_status`

### Step 4：注册 —— `src/tools.ts` 的 `ToolRegistry`

3 个 Tool 被存入 ToolRegistry，AI 可以随时查、随时调。

### Step 5：AI 调用 —— `tools/call` 的完整流程

当 AI 说 "查一下 id 为 42 的宠物" 时：

```
1. AI → MCP Server:  POST /mcp
   {
     "jsonrpc": "2.0",
     "id": 2,
     "method": "tools/call",
     "params": {
       "name": "petstore_get_pet_by_id",
       "arguments": { "petId": 42 }
     }
   }

2. MCP Server 内部（src/tools.ts execute 方法）：
   a. 找到 petstore_get_pet_by_id 对应的 Operation
   b. 把路径模板 /pet/{petId} 里的 {petId} 替换成 42 → /pet/42
   c. 组装目标 URL：https://petstore.swagger.io/api/v3/pet/42
   d. 发 HTTP GET 请求过去

3. 目标 API 返回：
   {
     "id": 42,
     "name": "Buddy",
     "status": "available"
   }

4. MCP Server → AI:
   {
     "jsonrpc": "2.0",
     "id": 2,
     "result": {
       "content": [{
         "type": "text",
         "text": "{\"id\":42,\"name\":\"Buddy\",\"status\":\"available\"}"
       }]
     }
   }

5. AI 看到结果，告诉用户：
   "宠物 #42 叫 Buddy，状态是 available。"
```

## 第四部分：代码结构走读

### 4.1 依赖关系

```
@api2mcp/core      ← 最底层，不依赖任何其他包
    ↑
@api2mcp/server    ← 依赖 core
    ↑        ↑
@api2mcp/rag      @api2mcp/agent    ← 各自依赖 core；agent 额外依赖 rag
    ↑        ↑
@api2mcp/cli      ← 依赖上面所有，是唯一对外暴露的入口
```

`src/` 目录下的文件（`tools.ts`、`server.ts` 等）是独立版本，import 路径指向 `../packages/`，作用等价于 `@api2mcp/server` 包内对应的文件。

### 4.2 每个文件在做什么

#### `packages/core/src/types.ts` — 所有类型定义

定义了项目里用到的所有数据结构。核心类型：

- `ApiOperation` — 描述一个 API 接口（method + path + parameters + requestBody）
- `MCPToolDef` — MCP 工具定义（name + description + inputSchema）
- `ApiSource` — API 来源信息（名称 + 根地址）
- `ProxyRequest` / `ProxyResult` — 代理请求/返回的结构
- `ToolError` — 统一错误格式
- `AuthConfig` — 鉴权配置

```typescript
// 核心映射：ApiOperation → MCPToolDef
// 这就是整个项目做的最关键转换

// 输入（来自 OpenAPI 文档）：
interface ApiOperation {
  operationId: string;           // "getUserById"
  method: "GET" | "POST" | ...;  // "GET"
  path: string;                  // "/users/{id}"
  parameters: ApiParameter[];    // 参数列表
  requestBody?: ApiRequestBody;  // 请求体
  summary?: string;              // 一句话描述
}

// 输出（给 AI 看的）：
interface MCPToolDef {
  name: string;                  // "github_get_user_by_id"
  description: string;           // "Get a user by ID"
  inputSchema: {                 // JSON Schema
    type: "object";
    properties: { ... };
    required?: string[];
  };
}
```

#### `packages/core/src/parser.ts` — OpenAPI 解析器（152 行）

输入：OpenAPI JSON/YAML 文件路径
输出：`ApiOperation[]`

做的事情很直接：
1. 读文件 → 试 JSON.parse，失败就试 YAML.parse
2. 检查有没有 `openapi` 字段（验证确实是 OpenAPI 文档）
3. 遍历 `paths` 下的每个路径、每个方法
4. 提取 `operationId`、`method`、`parameters`、`requestBody`
5. 返回全部 Operation 的数组

#### `packages/core/src/converter.ts` — 转换器（115 行）

最核心的转换逻辑。一个函数 `convertOperation(op, source)`：

```
Operation                          MCPToolDef
──────────                         ──────────
operationId: "getUserById"  →      name: "github_get_user_by_id"
method: "GET"               →      (体现在 proxy.ts 里)
path: "/users/{userId}"    →      (体现在 proxy.ts 里)
summary: "Get a user..."   →      description: "Get a user by ID."
parameters: [...]           →      inputSchema.properties
requestBody.schema          →      inputSchema.properties（合并进去）
parameters 里的 required    →      inputSchema.required
```

命名规则：驼峰自动转蛇形
- `getUserById` → `get_user_by_id`
- `listRepoIssues` → `list_repo_issues`

#### `packages/core/src/contracts.ts` — 接口契约

定义了 4 个核心接口，所有实现必须遵守。这相当于「包之间的合同」：

```typescript
IOperationParser  — parse(source) → ApiOperation[]
IToolRegistry     — register() / list() / execute() / has()
IApiProxy         — execute(req) → ProxyResult
IRagStore         — index() / search() / clear()
```

#### `packages/server/src/server.ts` — MCP Server（108 行）

最简单的 HTTP Server。监听 `POST /mcp`，处理 JSON-RPC 请求：

- `initialize` → 返回服务器信息（协议版本、能力）
- `tools/list` → 返回所有注册的 Tool 定义
- `tools/call` → 从 registry 找到 tool → execute → 返回结果

没有用任何框架，就是 Node.js 原生 `http.createServer` + JSON 解析。

#### `src/tools.ts` — ToolRegistry 实现（92 行）

Tool 注册和执行的「仓库」：

```typescript
class ToolRegistry {
  // 存所有 tool：Map<toolName, { operation, source, def }>
  private tools: Map<string, ToolEntry>;

  register(op, source)  // 把 Operation 转成 MCPToolDef，存起来
  list()                // 返回所有 tool 定义（给 AI 看）
  execute(name, args)   // AI 要调某个 tool 时：
                        //   1. 找到对应 Operation
                        //   2. 拼接 URL（路径参数插值）
                        //   3. 注入 Auth Header
                        //   4. 发 HTTP 请求，返回结果
}
```

#### `packages/server/src/proxy.ts` — HTTP 代理（48 行）

真正发出 HTTP 请求的部分。用了 Node 原生 `fetch`：

```typescript
class ApiProxy {
  async execute(req: ProxyRequest): Promise<ProxyResult> {
    const url = this.buildUrl(req.url, req.queryParams);
    const response = await fetch(url, {
      method: req.method,
      headers: { 'Content-Type': 'application/json', ...req.headers },
      body: req.method !== 'GET' ? JSON.stringify(req.body) : undefined,
    });
    return { statusCode, headers, body: await response.json() };
  }
}
```

#### `packages/server/src/auth.ts` — Auth 管理（58 行）

从环境变量读取密钥，自动注入到 HTTP 请求头：

```typescript
class AuthManager {
  register("github", { type: "bearer", envVar: "GITHUB_TOKEN" });
  // → 发起请求时自动加上 Authorization: Bearer ghp_xxx

  getHeaders("github") → { Authorization: "Bearer ghp_xxx" }
}
```

支持的 Auth 类型：`bearer`、`api_key`、`oauth2`、`none`。

#### `packages/rag/src/store.ts` — 内存向量存储（82 行）

最简单的向量数据库实现：
- 存：文档 ID + 向量 + 元数据
- 查：用余弦相似度算「查询向量」与「文档向量」的相似度，返回 Top-K

```typescript
class MemoryVectorStore {
  add(doc)        // 存文档
  search(vec, k)  // 搜最相似的 k 个
  clear(spec)     // 清掉某个 spec 的所有文档
}
```

#### `packages/rag/src/indexer.ts` — 文档索引器（69 行）

把 Operation 变成可搜索的文档：

```
Operation { operationId: "addPet", method: "POST", path: "/pet", ... }
    │
    ▼  buildDocumentText()
"API: petstore. Operation: addPet. Method: POST /pet. Summary: Add a new pet..."
    │
    ▼  hashEmbedding()
[0.023, -0.087, 0.134, ...]  128 维向量
    │
    ▼  store.add()
存入 MemoryVectorStore
```

#### `packages/rag/src/retriever.ts` — 检索器（26 行）

用户问「怎么创建宠物」，RAG 系统：
1. 把问题也转成向量
2. 在存储里搜最相似的文档
3. 返回最相关的 API 接口列表（带相似度分数）

#### `packages/agent/src/orchestrator.ts` — Agent 编排器（145 行）

Agent 的决策循环：

```
while (步骤 < 10) {
  1. 给 AI 看所有可用的 Tool
  2. AI 决定下一步：
     - rag_search("创建 Issue") → 找到 github_create_issue
     - 或直接调 github_create_issue(owner, repo, title)
  3. 执行 Tool，拿到结果
  4. AI 看结果，决定是继续还是结束
}
```

提供了两个辅助 Tool：

- `list_available_tools` — 列出所有可用工具

#### `packages/cli/src/index.ts` — CLI 命令行入口（212 行）

## 第六部分：和同类项目的区别

| | 本项目 | openapi-mcp-gateway (Python) | criteo/openapi-to-mcp |
